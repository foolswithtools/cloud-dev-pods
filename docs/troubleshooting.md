# Troubleshooting

Real failure signatures with diagnostic + fix. Every entry below was hit during smoke testing or surfaced as a real bug we landed fixes for. If you hit something not listed, file an issue against `foolswithtools/cloud-dev-pods` (use the `bug.yml` template) with full repro steps + relevant log excerpts.

## bootstrap-aws.yml fails with `The security token included in the request is invalid`

**Cause:** the `AWS_BOOTSTRAP_ACCESS_KEY_ID` / `AWS_BOOTSTRAP_SECRET_ACCESS_KEY` secrets either don't match (different `create-access-key` invocations got mixed up) or got mangled by paste (extra whitespace, character corruption).

**Fix:** test the keys locally first, then push via stdin.

```bash
unset AWS_PROFILE
AWS_ACCESS_KEY_ID="<id>" AWS_SECRET_ACCESS_KEY="<secret>" aws sts get-caller-identity
# Expect the bootstrap user's ARN. If this fails, the keys are bad.

# If that worked, push via stdin (avoids GitHub UI paste-mangling):
printf '%s' "<id>" | gh secret set AWS_BOOTSTRAP_ACCESS_KEY_ID
printf '%s' "<secret>" | gh secret set AWS_BOOTSTRAP_SECRET_ACCESS_KEY
```

## bootstrap-aws.yml fails with `ENOENT ... config/config.yaml`

**Cause:** the upstream template `.gitignore`s `config/config.yaml`. Your fork has it locally but you haven't committed it, so the workflow runner doesn't see it.

**Fix:**

```bash
git add config/config.yaml
git commit -m "chore: initial config"
git push
```

Re-trigger `bootstrap-aws.yml`. Phase 12's `init-clone.ts` will eventually generate this file and prompt for a commit; until then it's manual.

## ECS task fails with `ResourceInitializationError ... AccessDenied ... secret:/cloud-dev-pods/oauth/...`

**Cause:** the Lambda's task definition specified a *partial* Secrets Manager ARN (without the AWS-generated 6-char suffix). ECS+IAM require the full ARN. This was Phase 9.7 — fixed upstream, but if you're on a fork that hasn't pulled the fix, sync upstream.

**Fix:**

```bash
gh workflow run sync-upstream.yml
gh run watch
# Merge the resulting PR.
gh workflow run cluster-up.yml         # redeploy PodManager Lambda with fix
```

## oauth2-proxy crashes with `cookie_secret must be 16, 24, or 32 bytes ... but is 44 bytes`

**Cause:** the auto-generated cookie secret was the wrong length. Phase 9.8 fixed the default upstream (passwordLength: 32), but existing deployments retain the original 44-byte value because secrets aren't regenerated by CDK on update.

**Fix (one-shot, manual):**

```bash
aws secretsmanager update-secret \
  --secret-id /cloud-dev-pods/oauth/cookie-secret \
  --secret-string "$(openssl rand -hex 16)"   # 32 hex chars = 32 bytes
```

Restart any running pods to pick up the new value.

## Pod URL returns 502 Bad Gateway

**Cause (transient):** the ALB target group is still registering. Health checks need ~30s after `pod-up` before traffic flows.

**Fix:** wait 30s and retry. If it persists past 2 min:

```bash
TG=$(aws elbv2 describe-target-groups --names "cdp-<pod-name>" \
  --query 'TargetGroups[0].TargetGroupArn' --output text)
aws elbv2 describe-target-health --target-group-arn "$TG"
```

If state is `unhealthy` for >2 min, check the pod's container logs:

```bash
aws logs tail /cloud-dev-pods/dev/pods --since 5m --filter-pattern "<pod-name>"
```

## Pod URL returns "No pod registered for this hostname"

**Cause:** the ALB listener has no rule matching that hostname. Either the pod was reaped, or pod-up is in flight, or the hostname doesn't match the pod's actual host header.

**Fix:**

```bash
gh workflow run pod-list.yml
gh run watch
```

If the pod isn't listed, it was reaped or stopped. Re-run `pod-up.yml`.

If it is listed but you still get the 404, check the listener rules:

```bash
LISTENER=$(aws ssm get-parameter --name /cloud-dev-pods/dev/cluster/listener-arn \
  --query Parameter.Value --output text)
aws elbv2 describe-rules --listener-arn "$LISTENER" \
  --query 'Rules[*].[Conditions[0].HostHeaderConfig.Values[0],Priority]' --output table
```

## Idle reaper stops fresh pods within ~5 minutes

**Cause:** Phase 9.8 bug — reaper queried CloudWatch with only `TargetGroup` dimension; AWS requires `(TargetGroup, LoadBalancer)` pair. Without the load-balancer dimension, queries returned 0 datapoints and the reaper interpreted that as "idle = stop."

**Fix:** sync upstream (Phase 9.8 is in `main`):

```bash
gh workflow run sync-upstream.yml
# Merge the PR.
gh workflow run cluster-up.yml      # redeploys IdleReaper Lambda with fix.
```

To temporarily disable idle-reap on a specific pod (e.g., for long builds):

```bash
gh workflow run pod-up.yml -f pod_name=long-job -f mode=browser -f idle_minutes=0
```

## `cluster-up` fails with "log group already exists" or "Table already exists"

**Cause:** an older deployment had `RemovalPolicy.RETAIN` on the VPC flow-logs CW log group (`/cloud-dev-pods/<env>/vpc-flow-logs`) or the DynamoDB `cloud-dev-pods-registry` table, so `cluster-down` left them behind. The next `cluster-up` then fails because CFN refuses to create a resource whose name is already taken.

**Fix:** the policy is now `DESTROY` upstream, but a stack deployed before the fix still has RETAIN baked into its CloudFormation template — you have to either redeploy the stack on the fix once before tearing down, or delete the orphans manually:

```bash
aws logs delete-log-group --log-group-name /cloud-dev-pods/<env>/vpc-flow-logs
aws dynamodb delete-table --table-name cloud-dev-pods-registry
# Wait for the table to fully delete before re-running cluster-up:
aws dynamodb wait table-not-exists --table-name cloud-dev-pods-registry
```

Then re-run `cluster-up.yml`.

## ALB rule limit exceeded (after ~95 browser pods)

**Cause:** AWS ALB has a hard default of 100 listener rules. With one rule per browser pod (host-header routing), you've hit the cap.

**Fix:** raise the quota (`L-EAD7E5BC` in Service Quotas). Default → 1000 takes 1–3 business days. Tunnel pods don't count (no ALB rule).

## Tunnel pod doesn't show in VS Code Desktop

**Causes (in order of likelihood):**

1. Wrong GitHub account on local VS Code. The `code tunnel` agent authenticated with the GitHub identity used during `pod-up`'s device-code flow. Local VS Code Desktop must use the same.
2. Device-code prompt timed out (default ~15 min) without you authenticating. Re-run `pod-up`.
3. ECS task crashed before `code tunnel` registered. Check `aws logs tail /cloud-dev-pods/dev/pods --filter-pattern <pod-name>`.

**Fix:** in VS Code Desktop, sign out of Remote Tunnels (`Cmd-Shift-P` → "Remote Tunnels: Sign out") and sign in with the same account you used in the device-code flow.

## EFS mount target not in same AZ as task

**Cause:** the task got placed in an AZ where EFS has no mount target. Should be impossible since `ClusterStack` creates mount targets in all private subnets, but possible if you customized the network in `infra/extensions.local.ts` and added subnets without mount targets.

**Fix:**

```bash
EFS=$(aws ssm get-parameter --name /cloud-dev-pods/dev/cluster/efs-id \
  --query Parameter.Value --output text)
aws efs describe-mount-targets --file-system-id "$EFS"
# Should be one mount target per private subnet.
```

If missing, redeploy `cluster-up.yml`. If you customized via extensions, ensure mount targets are created for any subnets that pods will land in.

## OAuth App won't accept callback URL with wildcard

**Cause:** GitHub OAuth Apps don't support wildcard subdomains in the callback URL field. (GitHub *Apps* do — it's a different product.)

**Fix:** for the smoke test or single-pod workflows, register the OAuth App with the specific pod hostname (`https://hello.pods.example.com/oauth2/callback`). For multi-pod, you need one OAuth App per pod, or migrate to a centralized oauth2-proxy with a single callback. Tracked as Phase 9.6+ polish.

## `cdk bootstrap` missing

**Cause:** the CDK toolkit hasn't been bootstrapped in this AWS account/region. `bootstrap-aws.yml` runs `cdk bootstrap` once per account/region, but if it never ran or got cleaned up, subsequent deploys fail.

**Fix:** re-run `bootstrap-aws.yml`. It's idempotent.

## `Not authorized to perform sts:AssumeRoleWithWebIdentity`

**Cause:** the OIDC trust policy on `CloudDevPodsDeployerRole` (or `CloudDevPodsPodOpsRole`) doesn't match this repo's `sub` claim. Most common after a repo rename or transfer.

**Fix:** re-run `bootstrap-aws.yml` — it's idempotent and reapplies the trust policy with the current repo path. You'll need bootstrap credentials again temporarily; create a fresh IAM user as in `setup-cli.md`.

## Trivy fails the runtime image build with new CVEs

**Cause:** a new CVE landed against an apt package in the upstream `gitpod/openvscode-server` or `ubuntu:24.04` base.

**Fix:** the runtime Dockerfiles do `apt-get upgrade -y` to pull current Ubuntu security patches automatically. Rebuild:

```bash
gh workflow run build-runtime.yml
```

If the CVE is in a JS package bundled inside the upstream image (and shows in the *informational* trivy scan, not the gating one), there's nothing your fork can do — wait for upstream to bump and pull via Dependabot.

## sync-upstream PR has conflicts

**Cause:** you've modified a file that's `[merged]` in `.upstream-sync.toml`, and upstream changed it too in a way `git merge-file --diff3` couldn't auto-resolve.

**Fix:** the PR body lists conflicted files. Open each, find the `<<<<<<<` / `|||||||` / `>>>>>>>` markers, resolve manually, push to the PR branch. Then `npm run lint && npm run test && npm run synth` before merging.

If your customization should *never* be overwritten, consider refactoring it to use `infra/extensions.local.ts` (which is `[user]`) rather than editing `[tracked]` or `[merged]` files. See [`extending.md`](extending.md).

## release-please isn't opening a release PR

**Cause:** no `feat:` or `fix:` commits since the last release. release-please only triggers on those Conventional Commit prefixes.

**Fix:** check `git log --oneline main` for at least one `feat(...):` or `fix(...):` since the last `chore(release):` commit. `chore:` and `docs:` don't trigger.

## EFS filesystem retained across `cluster-down` (#29)

**Resolved by default in v0.2.0.** `ClusterStack` now provisions the EFS filesystem with `RemovalPolicy.DESTROY` (see [ADR 0007](adr/0007-efs-cluster-removal-policy.md)), so `cluster-down` tears it down with the cluster. To restore the v0.1.x retain behavior, set `efs.retainOnClusterDown: true` in `config/config.yaml` and re-deploy `cluster-up.yml` BEFORE running `cluster-down`.

**Legacy fix (forks still on v0.1.x or with an EFS already orphaned):** use the maintainer cleanup script —

```bash
scripts/maintainer/cleanup-orphan-efs.sh --mode efs           # dry-run
scripts/maintainer/cleanup-orphan-efs.sh --mode efs --apply
```

Or manually, if the filesystem really is orphaned (no live `CloudDevPods-Cluster` stack):

```bash
EFS=$(aws efs describe-file-systems --query "FileSystems[?Name=='cloud-dev-pods-dev'].FileSystemId" --output text)
aws efs describe-access-points --file-system-id "$EFS" \
  --query 'AccessPoints[].AccessPointId' --output text \
  | xargs -n1 aws efs delete-access-point --access-point-id
aws efs describe-mount-targets --file-system-id "$EFS" \
  --query 'MountTargets[].MountTargetId' --output text \
  | xargs -n1 aws efs delete-mount-target --mount-target-id
aws efs delete-file-system --file-system-id "$EFS"
```

## No EFS access-point reuse on `pod-up` (#30)

**Resolved by default in v0.2.0.** The pod-manager Lambda now looks up an existing per-pod access point by `Pod` tag on every `pod-up` and reuses its `accessPointId` + `PosixUser.Uid`, only allocating + creating when no AP exists. The AP itself is the canonical source-of-truth for the pod's UID; the DDB `posixUid` field is a cache while the pod is registered. See [ADR 0004](adr/0004-efs-per-pod-access-point.md) (Reuse semantics) for details.

**Legacy fix (forks on v0.1.x with accumulated orphan APs):** run the maintainer cleanup script. It groups APs by `Pod` tag, keeps the newest per pod-name, and deletes the rest. APs whose pod-name is in the live registry are skipped automatically:

```bash
scripts/maintainer/cleanup-orphan-efs.sh --mode aps           # dry-run
scripts/maintainer/cleanup-orphan-efs.sh --mode aps --apply
```

Manual single-pod sweep (only when the pod is **not** running — deleting an in-use AP will hang ECS task shutdown):

```bash
EFS=$(aws ssm get-parameter --name /cloud-dev-pods/dev/cluster/efs-id \
  --query Parameter.Value --output text)
aws efs describe-access-points --file-system-id "$EFS" \
  --query 'AccessPoints[?Tags[?Key==`Pod` && Value==`<pod-name>`]].AccessPointId' \
  --output text | xargs -n1 aws efs delete-access-point --access-point-id
```

## Lambda task-definition revision drift

**Cause:** every `cluster-up` (or pod-manager redeploy) registers a new revision of the `cloud-dev-pods` ECS task-definition family. Old revisions remain `ACTIVE` indefinitely, slowing console listings and complicating audit.

**Fix:** deregister stale revisions, keeping the latest. The current task family `LATEST` is safe; older `ACTIVE` revisions can be deregistered without affecting running tasks (they keep their pinned definition).

```bash
aws ecs list-task-definitions --family-prefix cloud-dev-pods --status ACTIVE \
  --sort DESC --query 'taskDefinitionArns[1:]' --output text \
  | tr '\t' '\n' \
  | xargs -n1 aws ecs deregister-task-definition --task-definition
```

## BootstrapStack stuck in `ROLLBACK_COMPLETE`

**Cause:** the first `bootstrap-aws.yml` run failed partway (commonly: insufficient bootstrap-user perms, or a pre-existing role with the same name). CloudFormation marks the stack `ROLLBACK_COMPLETE`, which is a terminal state — subsequent `cdk deploy`s will fail with `Stack ... is in ROLLBACK_COMPLETE state and can not be updated`.

**Fix:** delete the stack and retry. There's nothing to keep in a `ROLLBACK_COMPLETE` stack — it never finished provisioning.

```bash
aws cloudformation delete-stack --stack-name CloudDevPods-Bootstrap
aws cloudformation wait stack-delete-complete --stack-name CloudDevPods-Bootstrap
gh workflow run bootstrap-aws.yml -f confirm_account_id=<account>
```

If `delete-stack` hangs, check the CFN console for resources stuck `DELETE_FAILED` — usually an IAM role with attached policies that needs manual detachment.

## ACM certificate validation timeout

**Cause:** `cluster-up` requested an ACM cert with DNS validation, but the validation `CNAME` records weren't propagated to the public resolver in time (CFN waits ~30 min, then fails). Most often: Route53 zone is in a different account from the cert request, or the zone delegation NS records at the registrar still point at an old zone.

**Fix:** check the cert's pending validation records and confirm DNS:

```bash
CERT=$(aws acm list-certificates --query "CertificateSummaryList[?DomainName=='<base-domain>'].CertificateArn" --output text)
aws acm describe-certificate --certificate-arn "$CERT" \
  --query 'Certificate.DomainValidationOptions[].ResourceRecord'
# For each Name returned, verify it resolves:
dig +short _<token>.<base-domain> CNAME
```

If `dig` returns nothing, the record isn't published. Add it to the correct zone (next entry covers the cross-account case) and rerun `cluster-up.yml`.

## Route53 hosted zone in a different AWS account

**Cause:** your apex domain's Route53 zone lives in a different AWS account from the one running cloud-dev-pods. ACM cross-account DNS validation works (you just publish the validation `CNAME` in the zone's account), but the ALB **alias** record needs the zone in the same account as the ALB — alias targets are account-scoped.

**Fix:** delegate a subdomain (e.g., `pods.example.com`) into the cloud-dev-pods account and use that as `baseDomain` in `config/config.yaml`.

```bash
# In the cloud-dev-pods account:
aws route53 create-hosted-zone --name pods.example.com --caller-reference "$(date +%s)"
aws route53 get-hosted-zone --id <new-zone-id> --query 'DelegationSet.NameServers'

# In the apex-domain account, add an NS record set for `pods` pointing at the
# four nameservers above. Once the delegation propagates, ACM cert + ALB alias
# both work without cross-account hops.
```
